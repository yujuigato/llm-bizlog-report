Background

這個系統在解決什麼問題？

業務人員每天寫日報，但這些客戶洞察全部散落在非結構化文字中，沒有任何系統能自動整理成可行動的情報。

😣 過去的痛點

  • 17 萬筆日誌散落在 CRM，無法快速分析
  • 企業痛點靠人工整理，極度耗時且不一致
  • 無法識別哪些產業有共同痛點，業務無法複製成功模式
  • 訂單路徑分析需要大量人工比對才能看出規律

✅ 系統如何解決

  • LLM 自動萃取每家企業的 8 大屬性與 7 層銷售情報
  • ML 聚類識別 7 種法人類型，找出共同痛點模式
  • 四路資料整合生成痛需熱圖與業務推薦卡片
  • Sankey / Sunburst 圖追蹤 167.6 億訂單的成交路徑
How It Works

系統如何運作？

四個階段各自解決不同層次的問題，從原始日誌到可行動情報。

1

Phase 1：法人基礎屬性萃取

Gemini 2.5 Flash 讀取每家企業的業務日誌，萃取 8 大基礎屬性。

產業別、企業規模、數位化程度、決策層級、採購頻率、痛點急迫性、預算意願、競品使用情形
M

Phase M：法人類型聚類

Scikit-learn K-means 依屬性將企業分為 7 種法人類型（C0–C6），找出同類型企業的共同痛點模式。

相同法人類型的客戶通常有相似的卡關原因和需求，業務可以複製同類型的成功話術
2

Phase 2：L1–L7 七層銷售情報

LLM 深度分析每家企業，從表面現象一路萃取到隱性需求與解決方案線索。

L1 現象 → L2 猜測原因 → L3 客戶痛點 → L4 隱性痛點 → L5 需求目標 → L6 業務洞察 → L7 解決方案線索
3

Phase 3：四路整合與視覺化

將屬性、聚類、七層情報、問卷資料四路整合，生成痛需熱圖與個人化業務推薦卡片。

4

Phase 4：訂單時間序列分析

分析 167.6 億訂單的成交路徑與再購行為，以 Sankey 圖和 Sunburst 圖呈現客戶旅程。

Engineering Challenges

解決了哪些技術難題？

Challenge 1

17 萬筆 LLM 呼叫的成本控制

直接呼叫 Gemini 處理 17 萬筆會產生極高費用和時間，且日誌長短差異大。

↓ 解決方案

批次處理 + JSON 快取：相同企業的屬性不重複萃取；選用 Gemini 2.5 Flash（速度快、成本低）而非 Pro

Challenge 2

非結構化文字到結構化情報

業務日誌寫法千變萬化，同樣的痛點可能有完全不同的表達方式，直接萃取容易遺漏。

↓ 解決方案

設計階層式 Prompt：先萃取表面現象（L1），再逐層引導 LLM 深挖到 L7，每層都有明確的判斷標準

Challenge 3

聚類特徵的選擇

直接用原始文字做聚類效果差；用 LLM 萃取的屬性聚類，屬性定義的品質直接影響分群結果。

↓ 解決方案

Phase 1 的 8 大屬性設計以「業務行為差異」為準則，確保不同屬性組合的企業在業務策略上真的需要差異化對待

Challenge 4

訂單路徑的視覺化呈現

73,934 筆訂單的路徑資料如果用表格呈現，主管完全看不出規律；需要讓非技術背景的人一眼理解。

↓ 解決方案

Plotly Sankey 圖呈現產品組合路徑，Sunburst 圖呈現樹狀客戶旅程，互動式 HTML 讓主管可以自行探索
